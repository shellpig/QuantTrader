## [P2-D] [✅ 已處理 @ 451e2aa] 999.0 sentinel value 在 tearsheet 顯示時要轉成 N/A

- **位置**：`src/backtest/metrics.py` L62（sortino）、L67（calmar）、L87（profit_factor）
- **狀況**：當分母為零或無下方風險時，三個比率欄位回傳 `999.0` 作為 sentinel

## [P2-D] [✅ 已處理 @ 451e2aa] MA cross 訊號在資料起點若已處於金叉狀態會誤發 +1

- **位置**：`src/strategy/examples/ma_cross.py` L40-48
- **狀況**：`trend` Series 預設值為 `-1`。當 rolling 還沒滿（前 N-1 筆 `has_ma=False`）時 trend 維持 `-1`。第 N 筆首次 `has_ma=True`，若 `ma_short > ma_long`，trend 從 `-1` 跳到 `+1`，`diff==+2` 觸發 buy signal — 但這不是真正的金叉，只是 warm-up 結束的偽訊號
- **風險**：實盤接資料時，若資料起點剛好處於多頭趨勢（`ma_short > ma_long`），會在第 60 個 bar 誤發一筆 BUY；回測會把這筆當成 valid entry 計入績效
- **處理階段**：2-D（向量化引擎整合層）— 引擎跑回測時應丟掉前 `max(ma_short, ma_long)` 筆 warm-up 資料；或在 strategy 層改用 `trend.shift(1).notna()` 屏蔽首次有效 bar
- **發現於**：驗收 2-A
- **處理摘要**：`src/backtest/engine_vec.py:177-184` `_drop_warmup_signals(strategy, signals)` 在 `run()` 取得 signals 後立即呼叫，把前 `max(ma_short, ma_long)` 筆設為 0。Strategy 層保留原樣（policy 與 engine 解耦）。

### [P3-C] [✅ 已處理 @ 2258c27] 🚨 事件引擎沒建構 round-trip trades，導致 win_rate / profit_factor / max_single_loss 全壞

- **位置**：`src/backtest/engine_event.py` L74（`trade_records.append(fill.to_dict())`）+ L91（`_build_result(...)` 把 fill_records 直接餵給 `calculate_metrics`）
- **狀況**：事件引擎把每筆 `FillEvent` 的 dict 當成「一筆 trade」塞進 trade_records，但 `metrics.py` 期待的是 round-trip trade（含 `entry_date / exit_date / entry_price / exit_price / pnl`）。fill_dict 沒這些欄位，`calculate_metrics` 走 fallback 路徑後：
  - `total_trades = len(fills)` — BUY+SELL 各算一筆，是真實 round-trip 的兩倍
  - `pnl` 欄被建空 Series，所有筆 pnl=0
  - `win_rate = 0%`（沒 winning trade）
  - `gains=0, losses=0` → `profit_factor = 999.0`（被當無虧損誤判為「無敵」）
  - `max_single_loss = 0`
- **風險**：**所有用事件引擎跑出來的 tearsheet 績效報表都是錯的**。權益曲線正確（mark-to-market 有做），但 trade-level 指標全失真。3-D 端到端範例策略跑完看到的 win_rate/profit_factor 會嚴重誤導
- **處理階段**：**3-C（必修，不該推到 3-D）** — 引擎自己要追蹤未平倉部位、在 SELL 成交時建構 round-trip：
  1. 仿照 `engine_vec.py:82-110` 的 `_OpenTrade` dataclass 與 PnL 累加邏輯
  2. SELL fill 後計算 `pnl = (exit_price - entry_price) × qty - buy_cost - sell_cost`
  3. trade_records 改 append `{"entry_date", "exit_date", "side", "quantity", "entry_price", "exit_price", "pnl"}`
  4. 若同 symbol 有多次加倉，採 average-cost 配對（與 `SimpleAccount.cost_basis` 一致）或 FIFO；規格沒指定，但要與向量化引擎一致以便 3-D 對照
- **發現於**：驗收 3-C
- **建議補測試**：3-C 的 6 個 spec 測試只驗證「形狀」沒驗證「數字」。修這條的同時補：
  - `test_round_trip_pnl_matches_hand_calculated` — 6-bar fixture，BUY @ bar[1].open、SELL @ bar[4].open，逐欄 assert entry_price/exit_price/pnl

## 2026-04-26 Code Review 新增待修項目

### [P1] ParquetStorage 的 symbol 可造成路徑穿越寫檔

- **狀況**：`src/data/storage.py:81-88` 直接把 `symbol` 接進路徑，例如 `..\..\escape` 會讓輸出位置離開預期的 `data/raw/tw/{symbol}` 或 `data/processed/tw/{symbol}`。
- **風險**：若 `symbol` 來自 UI、API 或外部資料，可能造成任意檔案寫入或覆蓋非預期位置。
- **建議處理**：限制 `symbol` 格式，例如台股代號白名單 regex，並在 `resolve()` 後確認目標仍位於 `data_dir` 之下。
- **預計處理 phase**：近期安全修正。

### [P1] 事件引擎無法跑 1min / 5min 資料

- **狀況**：`src/backtest/engine_event.py:253-268` 只處理 pandas 舊頻率別名 `T`。目前 pandas 對分鐘資料可能回傳 `min` / `5min`，導致 `BarEvent.freq` 驗證失敗；`len(index) < 3` 時 `pd.infer_freq()` 也可能直接丟錯。
- **風險**：Phase 3 的分鐘 K 事件驅動回測會被阻斷，尤其是 1 分鐘與 5 分鐘資料。
- **建議處理**：支援 `min`、`5min`、`h` 等新 alias；對資料長度不足或無法推斷頻率時穩定 fallback 到 `1day` 或呼叫端指定頻率。
- **預計處理 phase**：Phase 3 事件引擎修正。

### [P2] pending order 會用下一根 bar 的 symbol 成交

- **狀況**：`src/backtest/engine_event.py:229-230` 建立 `FillEvent` 時使用 `match_ctx.symbol`，不是 `order.symbol`。多標的資料或策略送出非當前 bar symbol 的訂單時，訂單可能被記成下一列資料的 symbol。
- **風險**：多標的回測會把成交、稅率、持倉與 PnL 歸到錯誤標的，績效報表失真。
- **建議處理**：`FillEvent.symbol` 應使用 `order.symbol`；撮合前應確認目前 bar 的 symbol 與 order symbol 相符，不相符時延後、跳過或依多標的資料結構查找正確 bar。
- **預計處理 phase**：多標的支援前必修。

### [P2] 買進張數未預留交易成本

- **狀況**：`src/backtest/engine_vec.py:72-82` 用 `cash // price` 決定買進數量，之後才扣手續費與滑價；`src/strategy/examples/ma_cross.py:104-107` 的 event 路徑也只用現金與價格計算下單量。
- **風險**：向量化引擎可能出現負現金並形成不真實槓桿；事件引擎則可能在成交時因 `SimpleAccount` 現金不足而丟錯。
- **建議處理**：抽出共用 sizing helper，計算最大可買數量時把 commission、slippage 與最低手續費納入，並讓 vectorized / event strategy 使用同一套邏輯。
- **預計處理 phase**：Phase 3-D / 回測一致性修正。
